# PRD: Taste

## Summary

Build Taste, a FOSS artist recommendation system inspired by Gnoosic's simple input/output loop, but seeded from public knowledge graphs instead of requiring a large existing user base. The product should work as a CLI, embeddable library, and web app.

The core browser experience should stay close to Gnoosic's simplicity:

1. User enters 3 artists they like.
2. System shows one recommended artist prominently.
3. User responds: yes, no, or I do not know.
4. System uses that feedback to continue the recommendation session.

Power-user surfaces can expose more controls:

- CLI commands for import, recommendation, inspection, and graph debugging.
- Library/API access for music and tech users.
- Optional recommendation modes such as safe, adjacent, deep cut, influence, and descendant.
- Explanation views showing why each recommendation appeared.
- Future artist previews, such as links or embedded previews from external music services where legally and technically available.

Naming:

- Project: Taste
- CLI: `taste`
- Server daemon: `tasted`
- Optional personal/local alias: `tst`

## Problem

Most music recommenders depend on massive private listening datasets, opaque ranking systems, or platform lock-in. A small open project cannot start with millions of users or a complete song database.

The initial product should avoid that cold-start trap by recommending artists from an explainable graph built from public data, especially Wikipedia and Wikidata.

## Goals

- Recommend artists without requiring a huge song database.
- Seed the graph from public artist metadata, links, genres, labels, categories, and relationships.
- Use artists as the primary unit, not songs.
- Keep recommendations explainable and inspectable.
- Keep the web experience extremely simple by default.
- Support a portable Markdown vault format for human inspection and editing.
- Support small Markdown vault packs for scenes, labels, regions, or curated recommendation neighborhoods.
- Provide the same core engine through:
  - CLI
  - Library API
  - Web app
- Support both human-readable and JSON-first workflows.
- Allow future user feedback to improve the graph.
- Keep the first implementation small enough to finish.

## Non-Goals

- Full Spotify replacement.
- Audio analysis.
- Streaming, playback, or music licensing.
- Perfect global artist coverage at launch.
- User accounts in the first version.
- Large-scale ML training.
- Built-in AI or LLM recommendation features.
- Song-level recommendations in the first version.
- Built-in music playback in the first version.
- Requiring full local copies of massive music datasets.
- Requiring GitHub Actions, scheduled jobs, or hosted data pipelines.

## Core Idea

Create an artist graph:

- Nodes are artists.
- Edges are relationships between artists.
- Edge weights represent recommendation strength.
- Edge labels explain the relationship.

Example edge labels:

- `shared_genre`
- `shared_scene`
- `shared_label`
- `collaborator`
- `member_connection`
- `influence`
- `descendant`
- `wiki_link`
- `category_neighbor`
- `weak_context`

Recommendations are generated by walking the graph from the user's seed artists, ranking candidates, and applying mode-specific scoring. The default should optimize for a genuinely connected recommendation, not simply the most famous or most obscure related artist.

The system should avoid lazy popularity matches. For example, a Deftones-like query should not collapse into only the most obvious adjacent mainstream artist if the graph contains better-supported alternatives.

## Data Sources

Sources should be separated by role instead of treated as one generic music database.

### Tier 1: Core Identity

Use these to decide what an artist is and avoid entity confusion:

- Wikidata: first structured source for QIDs, sitelinks, genres, labels, locations, dates, external IDs, and MusicBrainz IDs.
- MusicBrainz: canonical music identity source for MBIDs, aliases, disambiguation, artist relationships, labels, and releases.
- Wikipedia: context source for page links, categories, prose, and infobox data.

### Tier 2: Relationship Signals

Use these to build artist-to-artist edges:

- Wikidata relationships.
- Wikipedia page links and categories.
- MusicBrainz artist relationships.
- Shared labels, members, scenes, and specific genres.
- Curated vault notes.

### Tier 3: Open Taste Signals

Use these later to make the graph feel more like taste and less like an encyclopedia:

- ListenBrainz datasets.
- ListenBrainz popularity/listen-count data.
- User feedback from recommendation sessions.
- Personal imports such as ListenBrainz profiles, Last.fm exports, playlist exports, or plain artist lists.

### Tier 4: Preview and UX Sources

Use these later for listening previews or outbound discovery links, not for the first ranking engine:

- MusicBrainz external URLs.
- Wikidata official websites and social links.
- Bandcamp, YouTube, SoundCloud, or Internet Archive search links where appropriate.
- Wikimedia Commons media where available.

Avoid making Spotify, Apple Music, YouTube, Last.fm, or any paid/auth-heavy API required for core recommendations.

### Source Priority

For identity resolution:

```text
MusicBrainz ID > Wikidata QID > Wikipedia title > fuzzy artist name
```

For fact confidence:

```text
MusicBrainz relationships
> Wikidata structured claims
> Wikipedia infobox
> Wikipedia prose/categories
> manual correction
```

For recommendation edge strength:

```text
direct relationship
> shared member/collaboration/label
> shared specific scene or genre
> Wikipedia page link/category
> broad genre
```

### Phase 1

- Wikipedia pages
- Wikidata entities

Extract:

- artist name
- aliases
- Wikipedia URL
- Wikidata QID
- genres
- labels
- associated acts
- members
- origin
- active years
- page links
- categories

### Phase 2

Potential future sources:

- MusicBrainz
- ListenBrainz
- user-submitted corrections
- imported personal listening data
- curated scene packs

Research direction:

- Music recommenders are commonly collaborative, content-based, or hybrid.
- Collaborative approaches can be strong with listening/user data but have cold-start and popularity-bias problems.
- Content and knowledge-graph approaches can work earlier and provide explanations, but need careful ranking to avoid bland matches.
- This project should begin as an explainable knowledge-graph recommender and later incorporate collaborative/listening signals where open data is available.

Recommended source roadmap:

```text
v0: Wikidata + Wikipedia + manual vault notes
v1: MusicBrainz identity, aliases, and relationships
v2: ListenBrainz co-listen edges and popularity priors
v3: personal imports and curated community vault packs
```

Default data strategy:

```text
small local graph
+ lazy expansion
+ aggressive caching
+ optional Markdown vault packs
```

Recommendations should run from local SQLite only. Network access should happen only during explicit import/expand commands.

The project should support two operating profiles:

```text
local/dev:
  small graph
  lazy expansion
  Markdown vault packs
  SQLite
  no required hosted workflows

prod/global:
  larger prebuilt graph
  many sources merged
  many packs merged
  optional bulk imports
  optimized read-only graph artifacts or server database
```

Production can use every useful source. Local development and normal personal use should not require every source.

### Existing Datasets and Projects to Evaluate

These sources may reduce the amount of graph construction needed from scratch:

- ListenBrainz similar-artists data: open similar-artist results derived from ListenBrainz listening data. Candidate source for taste edges.
- ListenBrainz datasets: useful for building co-listen artist edges and popularity priors offline.
- MusicBrainz database dumps: canonical artist identity, aliases, labels, releases, and relationship data.
- Million Song Dataset: includes artist-level terms, MusicBrainz tags, and precomputed asymmetric artist similarity relationships. Useful as a legacy bootstrap/evaluation source.
- Last.fm 360K dataset: user-artist-play tuples from Last.fm. Useful for offline collaborative filtering experiments if license/terms fit.
- HetRec 2011 Last.fm dataset: smaller user-artist listening, tags, and social graph dataset. Useful for toy experiments and testing algorithms.
- Semantic Artist Similarity dataset: small artist similarity benchmark with MusicBrainz IDs, DBpedia URIs, biographies, and top-10 similar artists.
- Troi / ListenBrainz recommendation tooling: useful reference implementation for playlist/recommendation pipelines, even if not used directly.

Evaluation criteria:

- License compatibility with FOSS redistribution.
- Artist identity quality and availability of MusicBrainz IDs.
- Whether the source provides artist-to-artist edges, not just song metadata.
- Whether the source can be processed offline.
- Whether it helps avoid popularity-only recommendations.
- Whether provenance can be preserved in the vault and SQLite graph.

Large datasets should be optional advanced imports, not required for normal use. The default product should not require users or maintainers to download huge dumps, run scheduled rebuilds, or maintain hosted data workflows.

Bulk imports are allowed for production builds:

```sh
taste import wikidata --bulk
taste import musicbrainz --dump ./musicbrainz
taste import listenbrainz --dataset ./listenbrainz
taste pack add ./packs/*
taste graph build --profile prod
```

These workflows should produce the same graph model as local/dev mode, just with more coverage.

## Review and Correction Workflow

The project should not include built-in AI calls or require any model provider. Review and correction should work through ordinary project surfaces:

- Markdown vault edits.
- JSON imports/exports.
- CLI inspection commands.
- CLI validation before database import.

The same workflow should be usable by humans, scripts, or external tools. The project should not care whether a correction came from a person, a script, a curated vault pack, or an external assistant.

Good review tasks:

- Normalize genre/category text.
- Classify whether a link is musically meaningful.
- Label relationship type from retrieved evidence.
- Detect overly broad weak matches.
- Generate or edit short explanations from stored evidence.
- Flag entity mismatches, jokes, memes, typos, and bad aliases.
- Decide whether evidence is strong enough to keep an edge.

Bad correction patterns:

- Freeform "recommend artists like X."
- Inventing artist facts without source evidence.
- Ranking without retrieved evidence.
- Creating new edges without source evidence.
- Treating a synthetic user's taste as ground truth.

Corrections should be structured JSON or vault edits and stored with provenance. The CLI should validate corrections before writing them into SQLite.

Example correction JSON:

```json
{
  "source_artist": "Boards of Canada",
  "target_artist": "Autechre",
  "relationship": "shared_scene",
  "confidence": 0.86,
  "verdict": "keep",
  "evidence": ["shared genre: IDM", "shared label: Warp"],
  "too_broad": false
}
```

Initial review CLI flow:

```sh
taste edges candidates --limit 50 --json > candidates.json
taste edges evidence --source "Boards of Canada" --target "Autechre" --json
taste edges review --file corrections.json
```

Review should operate like a normal edit/validate loop:

1. Ask the CLI for candidate edges.
2. Ask the CLI for evidence.
3. Write structured corrections or edit vault notes.
4. Run CLI validation.
5. Let the CLI import valid changes.

This keeps the C core simple and avoids any model dependency.

## Recommendation Modes

### Safe

Prioritize high-confidence nearby artists. Penalize obscurity and weak edges less aggressively than generic popularity.

### Adjacent

Recommend artists one scene or genre away from the seed set.

### Deep Cut

Prefer lower-popularity artists that still have meaningful graph support.

### Influence

Prefer artists that influenced or preceded the seed artists.

### Descendant

Prefer newer artists influenced by, adjacent to, or downstream from the seed artists.

## Scoring

Initial scoring can be simple and hand-tuned.

```text
score(candidate) =
  graph_proximity
+ relationship_weight
+ seed_coverage_bonus
+ mode_bonus
+ explanation_quality_bonus
- popularity_penalty
- weak_edge_penalty
- already_seeded_penalty
```

Important ranking rules:

- Avoid recommending the input artists.
- Avoid returning only globally famous artists.
- Prefer candidates connected to multiple seed artists.
- Prefer candidates with explainable evidence.
- Penalize broad-only matches like "both are rock bands."
- Penalize "too obvious" recommendations when more specific candidates have comparable evidence.
- Prefer relationship specificity over raw popularity.
- Do not confuse niche with good; deep cuts still need real graph support.

## Quality Bar

A recommendation is good when:

- It has a concrete musical, scene, label, influence, collaboration, or genre relationship to the seed artists.
- It is not supported only by a broad category.
- It can be explained from stored evidence.
- It avoids obvious popularity collapse when better connected candidates exist.
- It is plausible to a music nerd even if they personally do not like it.

A recommendation is bad when:

- It is connected only by a huge genre bucket.
- It appears because the artist is globally famous.
- It is a joke, meme, typo, or bad entity match.
- It ignores the combined seed set and only matches one artist weakly.
- The explanation sounds like generic filler.

## CLI Requirements

Initial CLI commands:

```sh
taste import wikipedia --artist "Cocteau Twins"
taste import wikidata --artist "Cocteau Twins"
taste expand "Cocteau Twins" --depth 2
taste graph build
taste recommend "Cocteau Twins" "Slowdive" "My Bloody Valentine" --mode deep-cut
taste explain "Cocteau Twins" "Bowery Electric"
taste graph inspect "Deftones" --json
taste feedback "Deftones" "Team Sleep" --rating good
taste pack add ./packs/shoegaze
```

The actual binary name can be decided later.

CLI output should support:

- readable text
- JSON via `--json`

The CLI is the first serious workbench. It should be good for:

- running recommendations
- debugging why an artist was recommended
- inspecting graph neighborhoods
- exporting evidence for review and correction
- importing normalized data from TypeScript tools
- expanding the local graph around seed artists
- installing small Markdown vault packs

## Library Requirements

Expose a small API:

```text
import_artist(source, artist_name)
build_graph()
recommend(seed_artists, mode, limit)
explain(source_artist, target_artist)
record_feedback(seed_artists, target_artist, feedback)
```

The library should not require the web app.

## Web App Requirements

First version:

- Input 3 artist names.
- Show one main recommended artist.
- Provide yes, no, and I do not know buttons.
- Optionally show a small ranked list/details view for power users.
- Show short explanations when requested.
- Let user mark:
  - good
  - bad
  - too obvious
  - too far
  - already know it
- Leave room for future artist preview links or embeds.

No accounts required in v1. Store feedback locally or in a simple database.

## Storage

Use two storage layers:

- Markdown vault: portable, human-readable source/interchange format.
- SQLite: compiled runtime index for fast CLI and web recommendations.

The Markdown vault should be transferable and rebuildable into SQLite for local use, production deployment, or sharing.

Start with SQLite for the first runtime implementation, but design imports/exports around the vault model.

Core tables:

- `artists`
- `artist_aliases`
- `artist_sources`
- `artist_facts`
- `edges`
- `edge_evidence`
- `recommendation_feedback`
- `raw_source_documents`

SQLite keeps the CLI and local web app easy. A server deployment can move to Postgres later.

Default import flow:

1. TypeScript importer fetches and normalizes Wikipedia/Wikidata data.
2. Importer writes normalized JSON files and/or Markdown vault notes.
3. C CLI validates and imports JSON/vault data into SQLite.

Raw Wikipedia/Wikidata responses should be cacheable for debugging and reproducibility, but the recommender should run from normalized facts and edges.

Network/data rule:

```text
recommend: local SQLite only
import/expand: network allowed, cache required
pack install: local files only
```

## Markdown Vault and Packs

The vault is an Obsidian-like graph format for artists, genres, labels, scenes, and sources. It should be easy to inspect, edit, diff, commit to git, and share.

Example structure:

```text
vault/
  artists/
    cocteau-twins.md
    slowdive.md
  genres/
    shoegaze.md
    dream-pop.md
  labels/
    4ad.md
  sources/
    wikidata-Q12345.md
```

Example artist note:

```md
---
type: artist
name: Cocteau Twins
wikidata: Q12345
musicbrainz: example-mbid
aliases:
  - Cocteaus
genres:
  - Dream Pop
  - Ethereal Wave
labels:
  - 4AD
associated:
  - This Mortal Coil
sources:
  - wikidata-Q12345
---

# Cocteau Twins

## Links

- [[Dream Pop]]
- [[Ethereal Wave]]
- [[4AD]]
- [[This Mortal Coil]]
```

Example pack structure:

```text
packs/
  shoegaze/
    pack.yml
    artists/
      slowdive.md
      lush.md
      ride.md
    genres/
      shoegaze.md
      dream-pop.md
    labels/
      creation-records.md
```

Example `pack.yml`:

```yaml
id: shoegaze
name: Shoegaze Starter Pack
version: 0.1.0
license: CC0-1.0
description: Small curated graph for shoegaze and adjacent dream pop artists.
sources:
  - wikidata
  - wikipedia
  - manual
```

Vault commands:

```sh
taste vault export ./vault
taste vault import ./vault
taste vault validate ./vault
taste vault build-db ./vault --db graph.sqlite
taste vault diff ./vault --db graph.sqlite
taste pack add ./packs/shoegaze
taste pack validate ./packs/shoegaze
```

Rules:

- SQLite is optimized for runtime queries.
- Markdown is optimized for portability and correction.
- The same graph should be rebuildable from the vault.
- Vault import should validate links, entity types, aliases, and required fields.
- Any vault edits must pass C CLI validation before database import.
- Packs are small shareable vault subsets, not full database dumps.
- YAML frontmatter is the structured source of truth inside Markdown files.

## Implementation Notes

Suggested architecture:

```text
core/
  C graph engine
  C ranking
  C explanation
  C SQLite storage

importers/
  TypeScript Wikipedia importer
  TypeScript Wikidata importer

cli/
  C command wrapper around core

web/
  C server-rendered web app using the same core API
  plain HTML/CSS
  optional tiny JavaScript or htmx for feedback actions
```

The first implementation should favor correctness and inspectability over clever ML.

## Stack Options

Preferred path:

- Core: C.
- CLI: C.
- Web app: C server-rendered HTML, similar in spirit to cgit.
- Storage: SQLite.
- Importers: TypeScript.
- Review/correction workflow through CLI and vault files.

TypeScript is allowed where it removes friction:

- Wikipedia/Wikidata API calls.
- HTML scraping or cleanup if needed.
- JSON normalization.
- Optional review/correction scripts.
- Future frontend enhancements.

The core product should still feel like a small, fast C program with plain files, plain HTML, and an inspectable SQLite database.

The portable project artifact should feel like a music knowledge vault: plain Markdown files that can be versioned, edited, shared, and compiled into SQLite.

## Milestone 1: Barebones Local Prototype

Deliverable:

- CLI-only prototype.
- SQLite database.
- Manual seed import for a small set of artists.
- Lazy expansion around seed artists.
- Simple graph build.
- `recommend` command.
- `explain` command.
- `graph inspect` command.
- Basic vault import/export or a clear stub format.
- Basic pack format or a clear stub format.

No AI or external model dependency required.

Success criteria:

- Given 3 artist names from the seed set, returns 5-10 plausible artists.
- Each recommendation has at least one explanation.
- Results are deterministic.

## Milestone 2: Wikipedia/Wikidata Import

Deliverable:

- Fetch artist metadata from Wikipedia/Wikidata.
- Extract links, genres, labels, associated acts, and categories.
- Build edges from extracted facts.
- Store provenance for imported facts and generated edges.

Success criteria:

- Can seed a general graph from a small but mixed artist set.
- Can explain which facts created each edge.
- Can export imported facts into a Markdown vault and rebuild SQLite from that vault.
- Can install a small Markdown pack and use it in recommendations.

## Milestone 3: Graph QA and Correction

Deliverable:

- CLI commands for candidate export, evidence inspection, and correction import.
- JSON-only structured corrections.
- Vault edit and validation workflow.
- Store verdict, confidence, relationship label, and rationale where corrections are applied.
- No direct AI or LLM dependency in the C core.

Success criteria:

- Corrections cannot create artists or facts without source evidence.
- Corrections can label retrieved evidence.
- Bad/generic edges can be filtered or downweighted.
- Users can run the review flow manually or with external tools.

## Milestone 4: Web App

Deliverable:

- Minimal local web app using the same core engine.
- Artist input.
- Mode selector.
- Recommendation list.
- Explanation view.
- Feedback buttons.
- Server-rendered HTML from C.

Success criteria:

- Web app can run locally against the same SQLite DB as CLI.
- User feedback is stored.
- Pages are usable without a large frontend bundle.
